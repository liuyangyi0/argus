# 零、先说最重要的一件事:定位转变

原 UX v1 的隐含前提是**用户在"管理这台摄像头"**——他在配置、在审查、在发布、在调参。这对工程师是对的,对刚上线阶段的操作员也是对的。

但一台摄像头稳定跑起来之后,系统进入**值班态**——值班员 99% 的时间只做一件事:**盯着一堵视频墙,等系统说话**。这个状态下,"摄像头详情页"不是主界面,"总览卡片页"也不是主界面。**主界面是视频墙**。

v1 的总览页把视频墙漏掉了,这是最大的结构性缺口。其他所有缺口(回放、降级提示、交接班)本质上都是**围绕"值班态"这个场景的功能**,v1 没有为值班态做设计,所以它们都没落地。

**这次增强的第一件事就是把"值班台"这个页面真正设计出来**,并把原来的"总览"降级为值班台的一个切换视图。信息架构仍然是五个顶层页面,但第一页的灵魂变了:

```
v1:  🏠 总览  → 卡片式健康面板(静态)
v2:  🖥️ 值班台 → Video Wall 为主,卡片为辅(实时)
```

这个转变定下后,其他六个缺口全部对号入座。我们开始逐个填。

# 一、告警回放(最大的漏洞)

## 1.1 问题陈述

v1 的告警详情页只显示**触发那一帧**的画面和热力图。出事之后值班员无法回答三个最基本的问题:

- **这个异常是突然出现的,还是缓慢堆积的?**
- **异常出现前 30 秒,画面上发生了什么?**(比如有人路过放下了东西)
- **告警之后这个物体消失了吗?还是一直在那?**

没有"倒带"能力,告警就只是一个定格照片,事后调查价值大打折扣。这个缺口在核电场景下不是 nice-to-have,是 must。

## 1.2 解决方案:每个告警都带录像带

每一路摄像头在推理时都维护一个滚动帧缓冲区,触发告警时系统**把这一段 ring buffer 固化到磁盘**,连同所有中间信号(Dinomaly 分数、Simplex 分数、CUSUM 证据累积、YOLO 检测框)一起落盘。这段固化数据就是这个告警的"录像带"。后续所有回放、事故分析都基于这段数据。

这一点必须写进需求(架构文档对应补一条 **FR-033 告警环形缓冲区持久化**)。

### 1.2.1 资源预算(硬约束)

这不是可以写一句话带过的细节——约束条件里硬写了 **Intel NUC / Jetson Orin NX + 每节点 2-4 路摄像头**,所以必须算清楚内存和磁盘。**别上来就 16 路**,16 路是多节点汇总的规模,单节点 4 路才是实际负荷。

**内存常驻**:

| 参数 | 值 |
|---|---|
| 单路帧率 | 5 FPS |
| 窗口 | 前 **60 秒** + 后 **30 秒**(不对称,见下) |
| 单节点摄像头数 | 4 |
| 单帧压缩后大小(JPEG Q85,720p) | ~80 KB |
| 常驻帧数 | 5 × 90 × 4 = 1800 帧 |
| **常驻内存** | **~140 MB** |

前后窗不对称是因为——事故诊断里"异常之前发生了什么"远比"异常之后怎么样"重要。前 60 秒能让值班员看到人员进入/放下物体的全过程;后 30 秒足够确认物体是否还在原地。前后都 60 秒是浪费。

**磁盘增长**:

按告警级别**差别化固化**,这是控制磁盘增量的关键:

| 级别 | 固化窗口 | 固化帧率 | 单条大小 |
|---|---|---|---|
| HIGH | 前 60s + 后 30s | 5 FPS(原速) | ~36 MB |
| MEDIUM | 前 60s + 后 30s | 2 FPS(降采样) | ~15 MB |
| LOW | 前 10s + 后 10s | 2 FPS | ~3 MB |
| INFO | 只存触发帧 + 热力图 | — | ~200 KB |

按"每节点 4 路 × 每天每路 5 条 MEDIUM + 0.5 条 HIGH + 10 条 LOW"的悲观估算:

- 4 × (5×15 + 0.5×36 + 10×3) ≈ **~500 MB/天**

边缘节点保留 30 天滚动归档大概 **15 GB**,NUC 可接受,Jetson 紧张但配外挂 SSD 可以。**30 天之前的证据自动降采样归档**(只保留触发帧 + signals CSV + 缩略图,视频删除),归档大小压到 30 天后 ~2 MB/条。这条规则要配一个设置项。

### 1.2.2 "录像带还在形成中"的 UI 状态

这是原稿漏的。告警触发时刻 T,后 30 秒的录像带要到 T+30s 才完整。**在这 30 秒内如果值班员打开了告警详情,界面不能假装录像带已经完整**。时间线右端显示一个录制中的指示:

```
 ─60s       ─30s       T 🔴       +15s       +30s
 ├───────────┼───────────┼──────●──⋯⋯⋯⋯⋯⋯┤
                                ↑
                          ⏺ 录制中 · 剩余 15s
```

未完成段用虚线表示,播放头可以进入但会显示"录制中,15 秒后可完整回放"。

**如果录制中又触发了新告警怎么办?** 两次告警共享 ring buffer——新告警的前窗可能和旧告警的后窗重叠,系统把它们链接(`linked_alert_id`)并在时间线上互相标注。不存两份。

### 1.2.3 LOW 级告警的简化策略

LOW 级告警的回放能力**不是必需品**——它们多数是噪声级别的事件,值班员一键处置就过了。所以:

- LOW 级只保留前 10s + 后 10s 的 2 FPS 固化(见上表)
- 回放面板仍然显示,但**没有多轨时间线**,只有视频 + 触发点
- 多轨时间线是 MEDIUM/HIGH 的能力

这个分级是现实的,不是偷懒。

### 交互要点

**一是双窗口布局与历史对照**。左大窗是回放主窗口,右小窗是**同摄像头的历史帧对照**。原稿写的是"从基线集选一帧",但**基线集是主动采样的帧集合,不是按时间索引的视频**——不能保证 14:22 这一刻有对应基线帧,即使有,光照和人员位置也不匹配。真正有用的对照是**同一台摄像头、同一时刻、不同日期**的历史帧,从 ring buffer 的长期归档里取。

右上角下拉切换对照时间:

- **昨天同一时刻**(默认)
- **上周同一日同一时刻**
- **本月上一周同一时刻**
- **最近一次无告警时段**
- **手动选择日期时间…**

**为什么不默认自动对照?** 因为"自动挑一个对照"很容易挑错,值班员一旦发现对照帧本身也有异常(比如上周那个时刻也有个保温棉反光),整个对照视图就崩了。**让值班员知道对照帧是哪一天的、自己可以切换,比系统假装智能更可信**。

如果该摄像头历史归档不足(刚上线不到一周),对照窗显示"历史数据不足,无法对照",不尝试伪造。

**二是五条轨道的时间线**。这是整个回放界面的灵魂。传统视频播放器只有一条时间轴,但告警回放的核心需求是**理解"为什么在这一刻触发"**,所以必须把所有输入信号同步画出来。五条轨道每一条讲一个故事:

- **Dinomaly 分数**告诉你 ML 通道怎么想
- **Simplex 分数**告诉你兜底通道有没有独立证实
- **CUSUM 证据**告诉你"为什么是现在触发而不是十秒前"——你会看到它从 0 慢慢爬到阈值
- **YOLO 人员段**是最重要的解释器:如果触发前有一段"人员存在"的灰块,值班员立刻知道"哦,有人路过时放下了东西"
- **操作员动作**是审计轨道,后续其他班组回看时能看到"这个告警之前谁处理过"

拖动播放头时,五条轨道同步高亮当前时刻。点击某条轨道上的峰值,播放头跳到那一帧。

**三是 DVR 控件**。`⏮ ⏪ ▶ ⏩ ⏭` 是标准播放控件,速度档 `0.25x/0.5x/1x/2x/4x`。**键盘快捷键支持专业视频剪辑的 J/K/L 范式**:

- `空格` 播放/暂停
- `←/→` 单帧步进(每帧 200ms)
- `J` 反向加速(重复按加倍)
- `K` 暂停
- `L` 正向加速(重复按加倍)
- `Home/End` 跳到开头/结尾
- `[` `]` 设置截取起止点(用于标记关键时段)

值班员不需要都会这些,但对专业用户(事故调查员)这套快捷键能把半小时的翻查压缩到五分钟。

**四是叠加层开关**。`🔥 热力`(异常热力图叠加)、`📦 框选`(YOLO 检测框显示)、`⊕ 同步`(对照窗同步播放)三个开关随时切换。默认只开热力图,叠太多会干扰判断。

**五是关键帧书签**。系统自动在时间线上标三个关键帧:**首次出现**(Dinomaly 分数首次越过背景噪声)、**证据超阈**(CUSUM 越过触发阈值)、**实际触发**(分发告警的那一帧)。值班员点击书签直接跳转。专业用户还可以手动 pin 任意时刻("这一帧很有意思,我要标记出来给下一班看")。

## 1.4 一个微妙但重要的设计

回放面板默认**不自动播放**。值班员打开告警详情时,画面停在触发帧上,而不是倒回去从头播。这是故意的——**值班员先看到的是"发生了什么"的定格证据,想看过程再点播放**。自动播放会让值班员错过触发帧本身。

这条原则和 Netflix/YouTube 相反,但在事故分析场景是对的。

# 二、实时视频墙(值班台主界面)

## 2.1 问题陈述

前面说过,v1 把这个页面彻底漏掉了。没有视频墙,值班员只能轮流点开每个摄像头的详情页手动查看,这在核电控制室是不可接受的。

## 2.2 布局:四种标准视图 + 一种焦点模式

值班台页面顶部是一个布局切换器:

```
┌─────────────────────────────────────────────────────────────┐
│ 🖥️ 值班台          [1×1] [2×2] [3×3] [4×4] [焦点]  [卡片] │
├─────────────────────────────────────────────────────────────┤
```

**1×1** 全屏单摄像头,用于深度观察(通常搭配键盘切换快捷键 `1/2/3/…` 直接切到对应摄像头)。
**2×2 / 3×3 / 4×4** 是标准网格,覆盖 4/9/16 路。
**焦点模式** 是 PIP 式:一个大窗口 + 周围一圈小窗口,点击小窗切换主角。这是核电控制室最常用的模式——**主显示器一路主监控,旁边八路保持可见**。
**卡片模式** 是 v1 原本的总览,作为对比视图保留。

布局选择保存在用户 profile,重启系统后恢复上次的视图。

## 2.3 单个视频 tile 的设计

每个 tile 不只是视频流,它是一个信息密度极高的小组件:

```
┌───────────────────────────────┐
│ ┌───────────────────────────┐ │  ← 顶部状态条(20px 高)
│ │ CAM-03 · v013     ● ⚠ 🔥 │ │     名称 / 模型版本 / 健康灯
│ ├───────────────────────────┤ │
│ │                           │ │
│ │                           │ │
│ │       [实时画面]           │ │  ← 主视频区
│ │                           │ │
│ │       5 FPS               │ │
│ │                           │ │
│ │                           │ │
│ ├───────────────────────────┤ │
│ │ ▁▁▁▂▂▃▃▄▄▃▂▁▁  🔴 2     │ │  ← 底部状态条(28px 高)
│ └───────────────────────────┘ │     30s 分数迷你图 / 本日告警数
└───────────────────────────────┘
   正常: 细灰边 | 警戒: 2px 黄边 | 告警: 3px 红边脉动
```

### 顶部状态条

- **名称 + 模型版本**:永远可见。模型版本小字——值班员一眼扫过知道这台跑的是新版还是旧版,回滚之后不会困惑。
- **健康灯组**:三个小图标,`●` 在线/离线、`⚠` 是否有降级(镜头污染、闪光干扰等)、`🔥` 是否有活跃告警。图标的颜色本身承载信息,值班员三秒扫完九路。

### 底部状态条

- **30 秒迷你分数图**:这是视频墙里最有信息价值的元素。它把过去 30 秒的 Dinomaly 分数画成一个高度 16px 的 sparkline。**值班员瞟一眼就知道这路是"平的"还是"在动"**——比盯画面快得多。
- **本日告警数**:`🔴 2` 或 `—`,点击跳转到该摄像头的告警列表。

### 边框

边框是整个视频墙的"一眼看到问题"的核心。三档:

- **正常**:细灰边,完全不吸引注意
- **警戒(黄)**:2px 黄边,不闪动,表示"这台有健康问题或者分数在上升,但还没触发告警"
- **告警(红)**:3px 红边 + 慢脉动(1Hz,幅度很小),**不要快闪**——快闪会诱发眩晕和操作员疲劳,这是核电控制室的硬禁忌

### 基本交互

tile 的基本鼠标交互(单击/双击/右键/拖拽)在 §2.5 里和"告警响应"一起详细说明。

```
│                      │
│ · 审查基线 (1)       │
│ · 推进发布 (1)       │
│ · 漂移预警 (1)       │
│ · 交接班 [开始 →]    │
└──────────────────────┘
```

**待处理告警段按严重度差别化处置**(这是第一轮审查后的关键调整):

- **LOW 级**:卡片上直接 `[确认]` `[误报]` 两个按钮,三秒处置,不离开视频墙。这类告警多是噪声(光斑、反光),值班员瞟一眼就能判断。
- **MEDIUM 级**:快速按钮保留,但点击后弹二次确认("标记为误报?此操作将记入审计日志")。防止连续点击的疲劳误操作。
- **HIGH 级**:**没有快速按钮**,只有 `[查看详情 →]`。HIGH 级告警在核电场景下必须查看完整证据(回放、多轨时间线、交叉验证),一键处置是纪律红线。进入详情页本身就是一种"减速带",强迫值班员至少看一眼回放。

这条规则写死,不开放为"班组可配置"——任何配置项都会被疲惫的人关掉,然后出事。

**系统状态段**是四个小点代表单节点四路摄像头(多节点场景下按节点分组)。点击任一点跳转到该摄像头详情。

**待办段**把 v1 总览页里的"待办三行"搬过来,放到视频墙侧栏。这是唯一的调整——v1 里这是总览的核心,v2 里总览已经降级,所以把待办搬到视频墙侧栏,保持"一眼知道该干什么"的原则。

## 2.5 告警触发时的视觉响应(替代原"自动聚焦")

**原稿写的"自动聚焦模式"被删除**。原因:即使默认关闭,"新告警自动抢占主视窗"在核电场景下是危险的——值班员正在专心看 CAM-07,CAM-03 一个误报把主视窗抢走,这叫"注意力劫持"。界面不应该替操作员决定注意力放在哪。

**替代方案**:新 HIGH 告警触发时,系统用**三个克制的信号**通知值班员,但**不动布局**:

1. **对应 tile 边框变红 + 慢脉动**(和原稿一样,1Hz 低幅度)
2. **侧栏待处理告警段顶部插入新卡片**,伴随一次柔和高亮(500ms 淡入淡出)
3. **视频墙顶部弹出一个不遮挡画面的横幅**:

```
┌─────────────────────────────────────────────────────┐
│ 🔴 新告警: CAM-03 HIGH       [切换到 CAM-03]  [×]  │
└─────────────────────────────────────────────────────┘
```

这个横幅是**建议**不是**强制**——值班员可以点"切换",也可以按 Esc 关掉继续看原来的画面。横幅 10 秒后自动消失但告警仍在侧栏可见。

**如果短时间内多条告警怎么办?** 横幅堆叠,最多显示 3 条,更多的折叠为"+N 条新告警"。绝不自动跳转,绝不重排布局。

### 交互:

- **单击 tile**:在视频墙内部选中,周围小高亮。
- **双击 tile**:切到 1×1 全屏单摄像头模式,此摄像头为主角。
- **右键 tile**:弹上下文菜单

```
┌──────────────────────────┐
│ 全屏查看                 │
│ 进入摄像头详情页          │
│ ──────────────────       │
│ 暂停报警(15 分钟) ▸     │
│ 确认锁定清除              │
│ ──────────────────       │
│ 静音音频告警              │
└──────────────────────────┘
```

- **拖拽 tile**:重新排列视频墙布局,释放时吸附到网格位。

## 2.6 音频告警

核电控制室的听觉告警必须非常克制。设计规则:

- **LOW**:不响
- **MEDIUM**:单次柔和提示音(类似短信通知,1 秒)
- **HIGH**:两次急促提示音(不是警报声,不是尖叫)+ 一次语音播报 "CAM-03 高级告警"(可配置关闭语音)
- **CRITICAL**(未来扩展):持续警报,直到人工确认

任何音频告警都可以用 `[全局静音 15 分钟]` 按钮临时禁用(带审计)。这是**核电控制室的刚需**——会议、上级视察、应急演练时必须能快速静音。

# 三、历史检索与相似搜索

## 3.1 问题陈述

v1 的告警详情页有一行"相似的历史案例",但这个功能**没有入口**——值班员平时想查"上次那种保温棉反光的误报是什么样"时无路可走,只能在告警列表里往下翻。

这个功能的技术底座已经在需求里了(Dinomaly2 基于 DINOv2 backbone,天然可以提取 embedding 做最近邻检索)。UX 要做的是把这个能力变成两个清晰的入口。

## 3.2 入口一:告警列表页顶部的全局搜索条

在 `告警` 顶层页面顶部,筛选条之上,加一个搜索条:

```
┌─────────────────────────────────────────────────────────────┐
│ 🚨 告警                                                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🔍 ┌─────────────────────────────────────────┬────────┐   │
│     │ 搜索告警... (支持自然语言、帧相似搜索)   │ [图片] │   │
│     └─────────────────────────────────────────┴────────┘   │
│                                                             │
│     建议: [保温棉反光] [遗留工具] [阴影误报] [CAM-03 本月] │
│                                                             │
├─────────────────────────────────────────────────────────────┤
```

**文本搜索**。支持几种模式混用:

- **自然语言关键词**:"保温棉"、"手套"、"夜班"、"误报"——系统在告警描述、操作员反馈备注、误报分类标签上做全文检索。
- **元数据过滤**:`camera:CAM-03`、`severity:HIGH`、`date:2026-03`、`outcome:false_positive`。这些是"进阶语法",但右侧下拉菜单能帮用户构造。
- **混合**:"CAM-03 保温棉 误报"会自动翻译成 `camera:CAM-03 keyword:保温棉 outcome:false_positive`。

搜索条下方的"建议"行是**基于当前用户历史**的快捷词:系统看到这个操作员经常查"保温棉反光",下次进来直接给他一键点击。

**图片搜索**。右侧的 `[图片]` 按钮打开帧上传对话框:

```
┌─ 按图搜索 ──────────────────────────────────┐
│                                             │
│  上传一张帧或从告警库选取:                  │
│                                             │

│  │                     │                    │
│  │   [拖拽或点击上传]   │                    │

│                                             │
│     或者:                                    │
│     [从告警库挑选]  [从摄像头实时截取]      │
│                                             │
│  范围: [所有摄像头 ▼]  时间: [最近半年 ▼]  │
│  相似度阈值: ●────────  0.75                │
│                                             │
│                 [取消]   [搜索]             │
└─────────────────────────────────────────────┘
```

"从摄像头实时截取"特别有用——值班员看到视频墙上某一路有奇怪的东西,**不确定是不是误报**,可以一键截取当前帧,系统立刻在历史库里找相似的画面,看看过去这种情况是真还是假。这是 v1 完全没想到的场景。

### 案卷(Dossier)机制

结果可以勾选加入"案卷"。案卷是一个**相关告警的集合**,用于事故调查、月度评审、模型训练反馈闭环。案卷本身可以命名、备注、导出整包证据。

案卷管理入口在 `告警` 页顶部的一个标签页"我的案卷":

```
📂 我的案卷
  ├ XX-2026-04 异物事故调查 (8 条)   进行中
  ├ 2026-Q1 保温棉误报汇总 (23 条)   已归档
  └ CAM-07 阴影问题 (15 条)          已归档
```

案卷是一个**持久化的调查容器**,不是临时筛选。这对核电场景下的长期事故追踪非常重要——一次异物事件可能调查数周,期间要反复加新线索。

## 3.4 入口二:告警详情页的"相似案例"区

v1 已经有这个区块,增强点在于:

- **可交互**:不只是列表,每个相似案例可以直接点击展开小预览(触发帧 + 结论)
- **明确告知结论**:每个案例旁边用大字标 `真实` 或 `误报`,并显示处理人和处理时间。这是"从历史中学习"的关键——不只是"像",还要告诉你"上次是什么结果"
- **一键加入案卷**:相似案例旁有 `[+案卷]` 小按钮

这样告警详情页的相似案例就从"装饰"变成了真正的决策辅助。

# 四、区域编辑器

## 4.1 问题陈述

v1 只写了一句"Canvas 多边形绘制"。但区域编辑是**操作员日常最高频的配置动作**——新摄像头上线要画,发现误报热点要调,结构变动要改。这个界面做得烂会直接拖累整个系统的可用性。

## 4.3 绘制工具(最小可用集)

**原稿列了多边形 / 矩形 / 笔刷 + Douglas-Peucker 简化 / 魔棒 flood-fill / 磁吸边缘。这次审查后砍到只保留多边形和矩形**。

理由:区域编辑**不是高频操作**——一台摄像头部署时画一次,之后几个月可能不改。笔刷和魔棒虽然看起来酷,但前端实现要么用 OpenCV.js 在浏览器里跑(体积膨胀),要么后端 API 往返(交互延迟差)。在边缘节点有限的工程预算下这是过度设计。

### 多边形工具(主力)

- 单击添加顶点,双击或按 Enter 封闭多边形
- 绘制过程中按 `Esc` 取消
- 封闭后,每个顶点变成可拖拽的小圆点
- 拖顶点移动,右键顶点删除
- 在边上 `Alt+单击` 插入新顶点
- `Shift+拖动` 约束到水平/垂直
- `Ctrl+拖动` 整体平移多边形

多边形工具 + 足够多的顶点可以画任何形状,只是手动一些。核电场景下"慢一点但可控"比"快一点但不可控"重要。

### 矩形工具

快速绘制规则区域,对角拖动。完成后自动转为四顶点多边形(这样可以继续用多边形工具精调)。

### 从其他摄像头克隆(替代高级工具)

工具栏底部 `📋 克隆其他摄像头` 按钮:选择一台已配置好的摄像头,把它的区域复制过来。两台摄像头之间有标定好的单应性矩阵时自动投影;没有则按像素坐标直接复制。

这是"新摄像头部署时借用模板"的主要路径,节约 80% 的初始配置时间——**这才是实际高频的复用场景**,比魔棒有用得多。

### 不做的工具(v2.2+ 考虑)

- 笔刷 + Douglas-Peucker 自动简化
- 魔棒(flood-fill 边缘检测)
- 边缘磁吸(Canny 边缘图)
- 网格吸附

这些只有在"多边形工具被证明确实不够用"的生产反馈出来后再加,而不是一开始就堆。

## 4.4 多区域管理

右侧区域列表显示所有区域,可以:

- **勾选显示/隐藏**(不是启用/禁用,只是画布显示)
- **点击选中**,进入编辑该区域
- **拖拽排序**(影响 z-order,上层区域压过下层)
- **右键菜单**:复制、删除、禁用、导出 JSON

**区域类型冲突规则要可视化**:

- Include 区显示为**绿色半透明填充**
- Exclude 区显示为**红色半透明填充**
- Include 和 Exclude 重叠的部分显示为**红色斜线**(表示此处不会检测——Exclude 优先级更高)
- 用户一眼看到"我画的 Include 被这个 Exclude 咬掉一块"

## 4.5 属性面板细节

每个区域选中后在右下方显示属性:

- **名称**:可编辑
- **类型**:Include / Exclude
- **优先级**:CRITICAL / STANDARD / LOW_PRIORITY(对应需求 FR-009 的报警分级乘数)
- **阈值覆盖**:滑块,如果启用则覆盖全局阈值。滑块下方实时显示"预估告警频率: 0.4 次/天"(用历史数据反推)
- **生效时段**:24小时 / 仅白天 / 仅夜间 / 自定义(带小时选择器)。这对"白班门口经常有人出入但夜班必须干净"的场景是必需的
- **冷却时间**:同一告警在该区域的重复抑制时间
- **备注**:自由文本(进入审计日志)

## 4.6 预览模式(关键设计)

修改了区域之后**不直接生效**。点击 `[预览]` 按钮进入预览模式:

```
┌─────────────────────────────────────────────────────────────┐
│ 🔬 预览模式 · 使用待保存的区域配置运行检测 30 秒            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│         [实时画面 + 待保存区域叠加 + 预览告警]               │
│                                                             │
│  预览统计:                                                  │
│  · 已处理: 152 帧                                           │
│  · 触发告警: 0                                              │
│  · 与旧配置分歧: 2 处(旧配置会触发的位置,新配置没触发)    │
│                                                             │
│  ⓘ 预览模式下告警不会真的发送                               │
│                                                             │
│              [继续预览]  [返回编辑]  [保存并应用]           │
└─────────────────────────────────────────────────────────────┘
```

**预览模式的核心价值**是"让用户看到改动的后果"。特别是"与旧配置分歧"这一行——用户知道自己改的区域多排除了两个原本会误报的位置,这是**验证改动是否达到目的**的唯一途径。

## 4.7 保存与审计

`[保存并应用]` 点击后弹最终确认:

```
┌─ 保存区域配置 ──────────────────────────┐
│                                         │
│  CAM-03 区域配置变更摘要:                │
│                                         │
│  · 修改: "走廊主体" 多边形               │
│  · 新增: "窗台区域" Include             │
│  · 删除: "楼梯口" Exclude               │
│                                         │
│  变更原因 (必填):                        │
│  [新增窗台导致夜间误报,增加排除]       │
│                                         │
│  签字: [   wang   ]                     │
│                                         │
│            [取消]   [保存]              │
└─────────────────────────────────────────┘
```

每次保存都记入审计日志,包含**变更前 JSON 快照 + 变更后 JSON 快照 + 原因 + 签字**。区域配置的历史可以回溯,甚至可以一键回滚到某个历史版本。

# 五、降级模式 UX

## 5.1 问题陈述

v1 没有考虑一件事:**系统不是总是完整运行的**。YOLO 可能挂了,Dinomaly 可能挂了,报警分发断路器可能打开了,RTSP 可能断了,存储可能快满了。v1 的界面在这些情况下要么沉默要么崩溃,都是核电场景的大忌。

**核电控制室的黄金法则**:如果什么东西坏了,界面必须**主动、响亮、持续**地告诉操作员——坏了什么、什么时候坏的、影响了什么、现在该做什么。

## 5.2 全局降级条

页面顶部有一个**保留槽位**,默认 0 高度,一旦有降级事件立刻展开为 48px 高的横条。位置固定在所有页面顶部(值班台、摄像头、告警、模型、系统),不可关闭:

```
┌─────────────────────────────────────────────────────────────────────┐
│ ⚠ 系统降级中: 主检测模型不可用,已切换到 Simplex 安全兜底。[详情]   │
└─────────────────────────────────────────────────────────────────────┘
```

颜色分三档:

- **🟡 黄色**:轻度降级,单个子系统出问题但整体可用(例如 YOLO 人员过滤挂了)
- **🟠 橙色**:中度降级,核心能力下降但未停机(例如 Dinomaly 挂了切到 Simplex)
- **🔴 红色**:严重降级,部分功能完全不可用(例如存储空间满无法写入告警)

多个降级同时存在时,取最高等级为条色,并显示 `⚠ 3 项降级` 字样,点击展开完整列表。

## 5.3 典型降级场景的 UX 文案

每种降级必须有**预先写好**的文案,不是代码抛异常信息。文案三要素:**发生了什么 / 影响是什么 / 现在怎么办**。

### YOLO 人员过滤模块挂了

```
┌───────────────────────────────────────────────────────────────────┐
│ ⚠ 降级: 人员过滤不可用                              2 分钟前开始   │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  发生了什么:                                                       │
│    人员检测模块(YOLO11n)无法加载,可能是模型文件损坏或 GPU 故障。 │
│                                                                   │
│  影响:                                                             │
│    检测继续运行,但人员经过的帧不再被自动遮罩/跳过。              │
│    人员活动频繁的摄像头(如主通道)误报率可能显著上升。            │
│                                                                   │
│  建议操作:                                                         │
│    1. 可以临时暂停高人流摄像头的告警(进入摄像头详情 → 暂停)     │
│    2. 通知工程师检查 YOLO 模型文件                                │
│    3. 已自动开工单: #T20260405-002                                │
│                                                                   │
│  [查看受影响的摄像头]  [查看技术日志]  [我已知晓,暂时忽略 1h]   │
└───────────────────────────────────────────────────────────────────┘
```

同时:

- 所有摄像头 tile 顶部状态栏显示 `🧍⚠` 小图标
- 摄像头详情页的 Pipeline Stepper 在"推理"节点显示黄色警告
- 告警详情页在证据列表里,原本显示"YOLO 人员过滤: ✓"的行改为"YOLO 人员过滤: ⚠ 降级期间"——让未来的审查者知道这个告警是在降级时生成的

### Dinomaly 主模型挂了(切换到 Simplex)

这是更严重的情况:

```
┌───────────────────────────────────────────────────────────────────┐
│ 🔴 降级: 主检测模型不可用,已切换 Simplex 兜底      6 分钟前开始   │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  发生了什么:                                                       │
│    CAM-03 的主异常检测模型 (dinomaly_head_v013) 推理失败。        │
│    系统已自动切换到 Simplex 帧差分兜底通道。                      │
│                                                                   │
│  影响:                                                             │
│    · 检测仍在继续,不会漏检明显的静态异物                         │
│    · 但灵敏度下降,难以检测细微/缓慢出现的异常                    │
│    · 期间产生的告警会被标记 "Simplex-only"                        │
│                                                                   │
│  建议操作:                                                         │
│    1. 考虑临时提高该摄像头的巡检频率                              │
│    2. 立即通知值班工程师                                          │
│    3. 已自动开工单: #T20260405-003                                │
│                                                                   │
│  [切换查看 CAM-03]  [查看技术日志]                                │
└───────────────────────────────────────────────────────────────────┘
```

**关键点**:

- 这种情况下 `[我已知晓,暂时忽略]` 按钮**不提供**——这是必须处理的,不能让值班员一键隐藏。
- 所有在降级期间生成的告警,在告警卡片和详情页**都有一个 `⚠ 降级期生成` 的标签**。这对事后复盘至关重要——"那次告警为什么没检测到细节?因为主模型挂了,只有 Simplex 在跑"。
- Simplex 兜底期间,视频墙的对应 tile 边框颜色变成**蓝紫色**(与"正常/警戒/告警"区分),值班员一眼看到"这路在跑兜底"。

### 报警分发断路器打开

```
┌───────────────────────────────────────────────────────────────────┐
│ 🟠 降级: 报警分发中断                               1 分钟前开始   │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  发生了什么:                                                       │

│    触发原因: Webhook 目标服务连续 5 次超时 (10s/次)。             │
│                                                                   │
│  影响:                                                             │
│    · 告警仍在正常检测和入库,不会丢失                             │
│    · 当前队列中积压 3 条待分发告警                                │
│    · DCS 上报、邮件通知暂时不会收到                               │
│    · 界面内显示正常                                                │
│                                                                   │
│  建议操作:                                                         │
│    系统将在 30 秒后自动尝试半开状态探测                           │
│    预计 3 分钟内自恢复                                            │
│                                                                   │
│  [立即重试]  [查看积压队列]  [查看故障目标]                      │
└───────────────────────────────────────────────────────────────────┘
```

**重要设计**:重点强调"不会丢失"和"界面内显示正常"。这两句是给值班员的**安抚**——让他知道他在 Argus 里看到的一切都是对的,只是外部通知临时断了。核电场景下,任何让值班员质疑"我看到的信息是不是完整的"都会引发严重焦虑。

### RTSP 断流

这是单摄像头级降级,不进全局降级条,只在对应 tile 和摄像头详情显示:

视频墙 tile:
```
┌───────────────────────────────┐
│ CAM-03 · v013       ● 🔄 🔥  │
├───────────────────────────────┤
│                               │
│                               │
│        [灰色画面]              │
│                               │
│     🔄 重连中 (3/10)           │
│     距上次帧: 12 秒            │
│                               │
│                               │
└───────────────────────────────┘
```

摄像头详情 Pipeline Stepper:
- "采集"节点变黄
- 悬停显示"RTSP 断流,指数退避重连中"
- 倒计时下次重试时间

### 基线过期 / 漂移超阈

```
┌───────────────────────────────────────────────────────────────────┐
│ 💡 建议: CAM-07 基线可能已过期                                     │
├───────────────────────────────────────────────────────────────────┤
│ KS 漂移检测显示 CAM-07 的分数分布在过去两周持续偏移。             │
│ 建议开启新一轮基线采集以保持检测精度。                            │
│                                                                   │
│   [前往采集]  [查看漂移详情]  [稍后提醒]  [忽略此建议]            │
└───────────────────────────────────────────────────────────────────┘
```

这是**蓝色信息条**,不是降级。区分好"系统坏了"和"系统建议你做点什么",两者的视觉语言必须不一样,否则降级条会变成狼来了。

### 存储空间不足

```
┌───────────────────────────────────────────────────────────────────┐
│ 🔴 严重: 证据存储空间不足 (剩余 3%)                                │
│ 影响告警环形缓冲区持久化和证据回放。                              │
│ [前往清理工具]  [查看存储详情]                                    │
└───────────────────────────────────────────────────────────────────┘
```

这种红色降级**不允许"暂时忽略"**,必须由运维处理掉才消失。

## 5.4 降级事件历史

所有降级事件都是**一类一等公民**的日志,在 `系统` 页下有专门的"降级事件"标签页,按时间倒序列出。每条记录包含:

- 开始时间 / 恢复时间 / 持续时长
- 降级类型 / 严重等级 / 影响范围
- 触发原因(技术日志摘要)
- 期间生成的告警列表(带"降级期生成"标签)
- 自动开出的工单链接

这个页面在**事故复盘**时是核心证据——"当时 Argus 为什么没报警?因为有这个降级",一查就知道。

## 5.5 原则总结

- **沉默是最大的罪过**:降级必须说话,主动说话
- **文案预先写好**:不要把技术异常信息直接抛给操作员
- **区分建议和故障**:蓝色是"可以做",黄/橙/红是"出事了"
- **留记录,不要静默恢复**:即使系统自愈了,降级事件也必须留痕

# 六、跨摄像头关联视图

## 6.1 问题陈述

需求 FR-027 明确写了"单应性投影 + 跨摄像头佐证",但 v1 的 UX 只在告警详情页写了一行"相邻摄像头同步触发 ✓"。这完全没发挥这个功能的价值。

跨摄像头关联是**高置信度告警的核心证据**——当两路不同视角同时看到异常,误报的概率急剧下降。这个证据必须被可视化得非常清楚。

## 6.2 入口一:告警详情页的跨摄像头子面板

在告警详情的"回放面板"之下,如果该告警有跨摄像头关联,显示一个新子面板:

```
┌─────────────────────────────────────────────────────────────────────┐
│ 🎯 跨摄像头佐证 (2 路同步触发)                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐                │
│  │             │  │             │  │             │                │
│  │   CAM-03    │  │   CAM-04    │  │  CAM-05     │                │
│  │  (主告警)   │  │  (佐证)     │  │  (无异常)   │                │
│  │             │  │             │  │             │                │
│  │  [缩略帧]   │  │  [缩略帧]   │  │  [缩略帧]   │                │
│  │             │  │             │  │             │                │
│  │  分数 0.91  │  │  分数 0.84  │  │  分数 0.12  │                │
│  │  ✓ 同步触发 │  │  ✓ 同步触发 │  │  — 未触发   │                │
│  └─────────────┘  └─────────────┘  └─────────────┘                │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │                                                             │  │
│  │        楼层平面图(俯视) + 摄像头视锥 + 异常定位            │  │
│  │                                                             │  │
│  │          ┌───┐                                              │  │
│  │          │ 03│──────╲                                       │  │
│  │          └───┘       ╲                                      │  │
│  │            │          ╲                                     │  │
│  │            │           ╲                                    │  │
│  │            │            ★ ← 异常定位(14:22:41)            │  │
│  │            │           ╱                                    │  │
│  │          ┌───┐        ╱                                     │  │
│  │          │ 04│───────╱                                      │  │
│  │          └───┘                                              │  │
│  │                                                             │  │
│  │          ┌───┐                                              │  │
│  │          │ 05│  (视角不覆盖此区域)                          │  │
│  │          └───┘                                              │  │
│  │                                                             │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  ⏱ 所有视角同步播放器 (共用主回放的时间线)                          │
│                                                                     │
│  [CAM-03 主视角]  [CAM-04 侧视角]  [楼层俯视]                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 设计要点

**一是三卡并列展示相关摄像头**。不只是"触发了的",还显示"视角内应该能看到但没触发的"(例如 CAM-05)——这是**反向证据**,也很重要。如果 CAM-05 能看到同一位置但没触发,值班员可能怀疑其中一路有问题。

**二是空间定位视图(三档降级)**。原稿假设所有部署都有楼层平面图,这是错的——**核电厂往往不会给部署方厂房平面图的原始数据**,那是保密的。所以这个视图必须有三档降级:

- **完整版(有平面图 + 单应性标定)**:上面 ASCII 草图展示的样子——真实楼层底图 + 摄像头视锥 + 异常点通过单应性投影到平面坐标。这是最理想状态。
- **抽象版(无平面图,有单应性)**:只画摄像头之间的拓扑关系图,每台摄像头一个节点,有单应性关联的摄像头之间连线。异常点在对应节点旁用小图标标出。值班员看不到"在厂房哪里",但能看到"哪几个视角同时发生了"。
- **最小版(两者都无)**:只显示三张缩略图横排,不画任何空间关系。文字说明"该部署未配置空间关联信息"。

系统自动按配置能力选择哪一档,**绝不伪造空间信息**——没有单应性就不要猜投影坐标。

**三是同步播放**。三个摄像头的回放时间线是锁定的,拖动主回放的播放头时,其他两路同步移动。可以切换"谁是主视角"。这一条在三档降级里都保留,和空间视图独立。

## 6.3 入口二:值班台的地图模式

值班台的布局切换器里加一个新选项 `[地图]`,**仅在部署配置了平面图或至少单应性关联时显示**,否则这个选项灰掉并悬停提示"需要部署时配置"。

```
┌─────────────────────────────────────────────────────────────┐
│ 🖥️ 值班台     [1×1] [2×2] [3×3] [4×4] [焦点] [卡片] [地图] │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  完整版(有平面图):                                         │
│         楼层平面图 + 所有摄像头点位                          │
│                                                             │
│  抽象版(只有单应性):                                       │
│         拓扑图:节点=摄像头,连线=单应性关联                │
│         按节点分组排列(不反映真实位置)                     │
│                                                             │
│  状态编码(两种版本共用):                                    │
│    ● 绿 = 正常 / ⚠ 黄 = 警戒 / 🔴 红 = 告警                 │
│    灰色连线 = 单应性关联                                     │
│    红色连线 = 当前有跨摄像头相关告警                         │
│                                                             │
│    点击摄像头点位 → 侧边弹出该路实时预览                     │
│    点击连线 → 展开关联告警列表                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**地图模式的部署要求要写进文档**:

- **完整版**需要部署时提供:楼层平面图图片(PNG/SVG)+ 每台摄像头在平面图上的坐标 + 单应性矩阵(每台摄像头画面到平面的投影变换)
- **抽象版**只需要:每对摄像头之间的单应性矩阵(用于跨摄像头佐证)
- **最小版**(地图模式不可用):只需默认配置

配置这些数据在运维文档里有专门章节(部署指南 § X),UX 本身不提供平面图标定工具——这是一次性部署动作,用桌面工具完成更合适。

地图模式的价值在"多摄像头全景监控"——值班员一眼看到"全厂哪里有问题"。但这个价值的前提是配置到位,配置不到位时降级到视频墙网格模式,不强求。

## 6.4 跨摄像头告警在列表里的表现

告警列表里,如果一个告警是跨摄像头关联的,显示一个链条图标 🔗 + 关联告警数 `🔗×2`。展开后显示所有关联告警,并且**整个关联组可以一次性处理**——一键"确认全部真实"或"标记全部误报"。

这避免值班员分别处理 CAM-03 和 CAM-04 的两个告警(它们本质上是同一个事件的两个视角)。

### 设计要点

**一是 KS 曲线是主图**。一条连续的线 + 一条水平的阈值线,越过就是问题。极其直观。

**二是分布对比密度图**。这比 KS 单个数字更能说明"漂移是什么样的"——用户一眼看到"哦,分数整体右移了",这解释了为什么误报率上升。

**三是只描述事实,不猜测原因**(第一轮审查后的关键调整)。原稿让系统"自动诊断可能原因:季节光照 / 新增物体 / 镜头老化",这是**虚假的权威感**——系统其实不知道为什么漂移,只是看到分布变了。给出"可能原因"会诱导工程师相信系统诊断,跳过真正的实地调查。

**正确的做法**:系统只说"观测到了什么"(整体右移 0.08、高分密度 +15%),然后提供一份**调查清单**让工程师去核对。调查清单比系统诊断更有价值——它引导工程师做该做的事,而不是让系统替他思考。

这里其实和原稿第五条原则"把数学翻译成直觉"**有张力**:翻译太狠就变成误导。平衡点是:**描述观测可以翻译,推断原因不能翻译**。

**四是行动选项必须在调查之后**。原稿写"建议行动(默认选中启动采集)",这是不对的——**在不知道原因的情况下默认推荐任何行动都是错的**。v2.1 的行动选项改成"完成调查后选择",没有默认选中,四个选项并列。如果是镜头脏了,清洁一下就行,采新基线是浪费;如果是真的场景变化,才需要新基线。让工程师判断,别让系统替他判断。

## 8.1 问题陈述

核电控制室是典型的 24 小时三班制,**每一次交接班都是事故高发点**——信息传递不完整会导致下一班误判或漏处理。v1 完全没设计交接班流程,这是一个运维层面的大漏洞。

### 设计要点(v2.1 重点)

**一是系统数据标注"供参考"**。第 1、2 段顶部明确写"系统自动统计,供参考",不让值班员误以为这是官方数字。每一项右侧有"已核对"勾选框,**全部勾选完才能进入签字步骤**——技术上是前端禁用"生成交接班记录"按钮直到所有勾选框都点上。

**二是修正说明栏**。如果值班员不同意系统某一项统计(比如某条告警被算错了类别),可以写进修正说明。**系统原始数据和修正说明都保留**,审计时两个数字都能看到。不是改,是加注。

**三是真实性声明分两条**。第一条覆盖系统数据的核对责任,第二条覆盖笔记内容的原创责任。两条分开是因为法律语境下"我核对了数据"和"我亲自写了笔记"是不同的承诺。

**四是签字是真的**。不是仪式,是审计证据。一旦签字,这份交接班记录永久入库,不可修改(修正说明在签字之前写,签字之后不能再改)。

**五是手写笔记**(第 3 段):自由文本 + 待办清单 + 附加告警。自由文本是"软信息"(直觉、猜测、建议);待办清单是"硬任务",下一班必须处理。附加告警用 `#告警ID` 语法可以直接嵌入链接。这一段没有自动生成,完全手写。

**这个改动的核心精神**:系统从"替你偷懒"变成"帮你核对"。两者在责任划分上完全不同。值班员的工作量只增加了十几秒点勾选框的时间,但责任链条从此清晰。

# 十、新增三条交互原则

v1 总结了五条原则,这次增强再补三条,都是从"值班态"和"事故分析"场景逼出来的:

## 第六,值班态优先于管理态

默认界面、默认动作、默认视图,都必须假设用户是"盯屏等事"的值班员,而不是"管理配置"的工程师。工程师的工作流是副路径,藏在点击深处。

这条原则的具体体现:值班台默认视频墙、侧栏三段式只放值班员最需要的东西、告警卡片支持不进详情页的快速处置、视频墙音频告警极度克制。

## 第七,出事必须能倒带

任何触发告警的时刻,系统必须保留完整的前因后果录像,并以默认能力(不是高级功能)提供给值班员。多轨时间线、速度控制、关键帧书签——这些不是"如果有时间就做",是核电场景的硬要求。

这条的反面教训:v1 只有触发帧,连"告警前 10 秒发生了什么"都回答不了。

## 第八,降级必须说话

任何降级状态都必须**主动、响亮、具体**地告诉操作员。沉默的降级是核电场景最大的信任杀手——操作员质疑过一次"我看到的信息是否真实",就再也建立不起完全信任了。

这条原则的具体要求:降级文案预先写好(不是代码抛异常信息)、区分"故障"(黄/橙/红三档)和"建议"(蓝色信息条)、每种降级说清楚"发生了什么/影响是什么/现在怎么办"、降级事件进专门的历史日志可供事故复盘追溯。

这些都不是细节,是核心。

# 十一、非功能设计(第一轮审查后补)

原稿只画屏幕,没写**权限、国际化、无障碍、网络降级**。这几块在核电语境下不是可选项,必须显式写清楚。

## 11.1 权限矩阵

需求里 RBAC 有 admin / operator / viewer 三级。原稿所有动作都没说"谁能做",这是一个大漏洞——如果不在 UI 上显式禁用,值班员就能误改工程师才能改的东西。规则如下:

| 动作 | viewer | operator | engineer/admin |
|---|---|---|---|
| 查看值班台、告警列表、摄像头实时画面 | ✓ | ✓ | ✓ |
| 查看告警详情、回放、相似搜索 | ✓ | ✓ | ✓ |
| 确认告警 / 标记误报(LOW/MED) | — | ✓ | ✓ |
| 确认告警 / 标记误报(HIGH) | — | ✓ | ✓ |
| 创建/编辑案卷 | — | ✓ | ✓ |
| 提交交接班记录 | — | ✓ | ✓ |
| 全局静音音频告警 | — | ✓ | ✓ |
| 调整阈值(滑块) | — | — | ✓ |
| 编辑检测区域 | — | — | ✓ |
| 启动/暂停基线采集 | — | — | ✓ |
| 批准基线 | — | — | ✓ |
| 启动训练 | — | — | ✓ |
| 推进发布(影子→金丝雀→全量) | — | — | ✓ |
| 回滚模型 | — | ✓(紧急) | ✓ |
| 清除异常锁定 | — | ✓ | ✓ |
| 修改用户/权限 | — | — | admin 专属 |
| 系统备份恢复 | — | — | admin 专属 |

**关键设计规则**:

- **无权限动作的按钮不隐藏,而是灰掉 + 悬停提示**"需要工程师权限"。隐藏会让值班员困惑"这个功能去哪了",灰掉可见但不可点可以解释。
- **回滚例外**:操作员(值班员)拥有紧急回滚权限。这和原稿的"回滚是零摩擦恢复性操作"原则一致——出事时不能让值班员等工程师上线。回滚动作记入审计日志,工程师次日复核。
- **权限变更记入审计日志**,admin 的每次授权变更都可查。
- **UI 加载时一次性获取当前用户的权限集**,而不是每个按钮点击时再查——减少后端压力,同时避免"按钮看起来能点但点了报错"的糟糕体验。

## 11.2 国际化(i18n)

需求 NFR 里把 i18n 列为"未来扩展",但 UX 层面如果一开始就硬编码中文,**后期翻译成本会翻倍**。v2.1 的硬性要求:

- **所有界面字符串走 i18n 框架**(Vue i18n),不允许在组件里硬编码。这一条 v2.0 就必须遵守,即使只发布中文版
- **预留中英双语**(`zh-CN` / `en-US`),其他语言作为 v2.2+ 扩展
- **时间/日期格式本地化**:`dayjs` + locale,不要手拼字符串
- **数字/单位本地化**:"1.25 次/h" 在英文里是 "1.25/hr"

**中文特有的设计要重新审视**:

- "王工"这种敬称在英文里没有对应,签字框的占位符应该是**当前用户的登录名**,不是"王"
- "白班/夜班/凌晨班"英文可以是 "Morning / Evening / Night" 或 "Shift A / B / C",由部署方在配置里选
- 报告 PDF 模板需要维护中英两份(不是运行时翻译)

**文档约定**:本 UX 文档里所有中文 UI 字符串都视为"示例",实际实现时走 i18n key,不直接复制字符串。

## 11.3 无障碍(Accessibility)

核电控制室的值班员里可能有**色盲**——约 8% 的男性有某种色觉异常。原稿大量依赖"绿/黄/红"三色传递状态,色盲用户完全看不出区别。这在**合规语境下不是可选项**。

### 硬性要求

- **所有颜色编码必须配合形状/图标冗余**:
  - 绿色 正常 → 加 `●` 实心圆 或 `✓`
  - 黄色 警戒 → 加 `⚠` 三角 或 `◆`
  - 红色 告警 → 加 `🔴` 或 `✗`
  - 降级条的黄/橙/红三档除颜色外还有不同的前缀图标(`ⓘ` / `⚠` / `⛔`)
- **必须通过色盲模拟测试**:
  - 用 Sim Daltonism 或 Chrome DevTools 的视觉障碍模拟,跑 deuteranopia(红绿色盲)、protanopia(红盲)、tritanopia(蓝黄色盲)三种模式
  - 测试时每个页面都要确认状态信息可辨
- **文字对比度达 WCAG AA**:
  - 正文 4.5:1
  - 大字(18pt+ 或 14pt+ 粗体) 3:1
  - 暗色主题是 Ant Design Vue 的默认暗色,但自定义颜色必须手动检查

### 其他无障碍

- **键盘导航**:所有交互都可以用 Tab / Shift+Tab / Enter / Space 完成,不强制鼠标。视频墙可以用方向键切换 tile。
- **焦点指示清晰**:当前焦点元素有明显的边框/底色,不是默认浏览器细虚线
- **屏幕阅读器标签**:关键按钮和状态元素有 `aria-label`,图表有文字 fallback 描述
- **字号可调**:支持浏览器缩放到 200% 不破版
- **音频告警可替代**:所有音频提示都有对应的视觉提示(红边脉动、横幅),听障用户不会遗漏告警

**这一节是硬交付要求,不是"有空再做"**。

## 11.4 网络降级场景

需求约束写了"核电站网络环境(可能无外网访问)"和"可能的弱网环境"。原稿只覆盖了"子系统挂了"的降级,没覆盖"网络挂了"的场景。

### WebSocket 断线

前端和后端通过 WebSocket 推送四个主题(health / cameras / alerts / tasks)。WS 断了值班台就"冻住"了——画面还在但数据不更新。必须显式显示:

```
┌─────────────────────────────────────────────────────┐
│ 🔄 实时连接中断,正在重连...  (上次更新: 12 秒前)   │
└─────────────────────────────────────────────────────┘
```

- 这个横幅在 WS 断开超过 5 秒时出现(避免瞬断闪烁),位于降级条之下
- 重连用指数退避,最多 10 次
- 重连成功后横幅自动消失,**不显示"已恢复"**(避免噪声)
- 重连期间所有"需要实时数据"的操作(确认告警、触发采集)按钮灰掉 + 悬停提示"连接中断,请等待恢复"

### 相似搜索 / embedding 计算

如果相似搜索依赖后端重跑 embedding,弱网或拥塞下延迟会很高。规则:

- 搜索请求 > 3 秒未返回显示"搜索中..."
- > 10 秒未返回允许取消
- 失败时显示"本次搜索失败",不是抛异常栈
- 有一个轻量级 fallback:"基于元数据的搜索"(仅用告警描述/标签/时间/摄像头),这个不需要 embedding,弱网下可用

### 归档盘不可达

"复制到归档盘"按钮如果归档盘是网络盘:

- 按钮点击后先**异步探活**(1 秒超时)
- 不可达时按钮灰掉并显示"归档盘不可达,请检查网络或选择本地下载"
- 不要让用户等 30 秒然后报错

### 邮件/Webhook 不可达

这种情况已经在"报警分发断路器"章节覆盖,不重复。

### 完全离线(内网隔离)

- 所有依赖 CDN 的前端资源必须本地化(Ant Design Vue、字体、图标)——不能有任何 `<link href="https://cdn.*">`
- 所有依赖外部 API 的功能(例如 Tile map 底图)要么关闭,要么有本地替代
- 文档中明确标注"哪些功能依赖外网",部署方可以选择开关

## 11.5 边缘资源预算

已经在 §1.2.1 里对 Ring Buffer 做了详细预算。这里汇总一下单节点(Intel NUC / Jetson Orin NX,4 路摄像头)的**总体资源预算**:

| 资源 | 预算 | 去向 |
|---|---|---|
| 内存(常驻) | ~1.5 GB | YOLO ~400MB · Dinomaly ~500MB · Simplex ~50MB · Ring buffer 140MB · 前端/API/DB ~400MB |
| 内存(峰值) | ~2.5 GB | 训练期间额外 ~1GB |
| 磁盘(系统+模型) | ~10 GB | 系统 + 模型仓库 + 基线若干版本 |
| 磁盘(30 天证据滚动) | ~15 GB | 按 §1.2.1 的差别化固化估算 |
| 磁盘(日志 + 数据库) | ~5 GB | 审计日志 + SQLite |
| **单节点总磁盘** | **~30 GB** | 留 50% 余量 → 推荐 64 GB SSD 起步 |
| CPU | < 70% 平均 | 推理 + 背景任务 + 前端服务 |
| GPU/NPU | 可选 | Jetson 上用 NPU 加速 Dinomaly,NUC 上 OpenVINO |

这张表必须在部署文档里写清楚,**不能让运维上线之后才发现磁盘满了**。

# 十二、优先级与落地建议(按第一轮审查后调整)

这六个增强不是同时做,建议的实施优先级(v2.2 调整):

**P0 必须 v2.0 就有**
1. **Ring Buffer 持久化与资源预算**(§1.2)——这是回放/跨摄像头视图的共同基础,不做其他都无从谈起
2. **告警回放:多轨时间线 + DVR**(§1.3)
3. **实时视频墙**(§2)——至少 1×1 / 2×2 / 焦点模式三种布局
4. **降级模式全局条 + 五种典型文案**(§5)
5. **告警处置按严重度分档**(§2.4)——这是合规红线
6. **权限矩阵**(§11.1)——基础 RBAC 必须 v2.0 就有
7. **无障碍硬性要求**(§11.3)——色盲适配不能后补

这七个不做,值班态根本跑不起来,或者过不了合规。

**P1 必须 v2.1 跟上**
8. **交接班流程(带人工复核)**(§8)
9. **区域编辑器最小可用集**(§4)——多边形 + 矩形 + 克隆 + 预览模式
10. **网络降级场景文案**(§11.4)
11. **i18n 框架接入**(§11.2)——即使只发中文版,框架也必须 v2.1 落地

**P2 v2.2 - v2.3**
12. 历史检索与案卷(§3)
13. 跨摄像头关联视图 + 地图模式(§6,含三档降级)
14. 漂移仪表板(§7,事实描述版)
15. 区域编辑器高级工具(笔刷/魔棒/磁吸)—— 仅在生产反馈确认需要后再做

**永远不做**
- 证据包导出(原 §2,产品决定不做此功能)
- 自动聚焦强抢主视窗(被审查否决)
- 漂移原因自动猜测(被审查否决)

# 尾声

这次增强的核心调整其实是一句话:**把 v1 的"摄像头管理工具"变成 v2 的"核电控制室值班台"**。所有具体的设计——视频墙、时间线回放、降级条、交接班——都是为这个定位服务的。

第一轮自审查之后,方案从"写得爽"变成"写得老实"——砍掉了过度设计(魔棒/笔刷/磁吸、自动聚焦、漂移原因猜测),加固了合规边界(HIGH 级无快速按钮、交接班强制复核),补齐了漏掉的非功能设计(权限、i18n、无障碍、网络降级、资源预算)。v2.2 进一步砍掉了证据包导出章节(产品决定不做)。

骨架到此应该是完整且诚实的。每一个小节都可以再下探一层到像素级别——时间线轨道的具体色彩规范、降级文案的字数上限、权限矩阵对应的前端实现模式、区域编辑器的鼠标状态机——告诉我先做哪个,我继续画到可以直接交付前端的颗粒度。
