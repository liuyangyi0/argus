# Argus 当前需求与范围说明

> 最后更新：2026-04-12

## 1. 产品目标

Argus 当前聚焦的目标如下：

- 使用固定摄像头持续检测场景异常、异物或异常状态变化。
- 在单节点、资源受限的边缘环境中稳定运行。
- 为操作员提供从发现、确认、回放到复盘的闭环处理流程。
- 为工程人员提供基线、标注、训练、发布和回滚能力。
- 在主模型不可用、流媒体异常或外部通道故障时保持降级可用，而不是整体停止服务。

## 2. 主要使用角色

当前系统涉及的主要角色包括：

- 操作员：查看总览、处理告警、查看回放、执行日常运维操作。
- 工程人员：管理基线、标注样本、创建与确认训练任务、切换模型、排查系统问题。
- 管理员：管理用户、审计、配置和备份。

需求描述默认围绕这三类角色展开，而不是抽象的“平台能力”。

## 3. 当前交付范围

### 3.1 视频采集与运行管理

- 支持 RTSP、USB、文件输入。
- 支持多摄像头并行运行，且每路摄像头有独立的生命周期与状态。
- 支持断线重连、状态查询、基础画质检查和健康信息汇总。
- 默认部署中，USB 摄像头路径需要与 go2rtc 配合使用，以保证浏览器访问与检测链共存。

### 3.2 检测管线

当前默认检测链要求包括：

- 支持 include / exclude 区域限制。
- 支持 MOG2 预筛。
- 支持 heartbeat 旁路，避免静态异物被背景吸收。
- 支持基于 YOLO 的人员或指定类别过滤。
- 支持异常检测主通道。
- 支持 Simplex 安全通道并行判断，并将其视为默认运行路径的一部分。
- 支持后处理、时序证据累积、自适应阈值和告警分级。
- 支持在模型异常、分割器异常或视频异常场景下进入降级状态，而不是直接中断全部服务。

### 3.3 告警与处置闭环

- 告警需要入库并可按条件筛选查询。
- 告警需要通过 WebSocket 向前端实时推送。
- 告警分发支持本地持久化，以及 Webhook、邮件等对外通道配置。
- 告警需要具备工作流状态，如待处理、确认、误报等。
- 告警需要具备回放取证能力，包括录像、关键帧、热力图、信号时间线和历史参考帧相关 API。
- 告警处理过程需要可追溯，便于审计和问题复盘。

### 3.4 基线、模型与训练

- 支持基线采集与版本化管理。
- 支持训练作业创建、查询、确认和拒绝。
- 训练作业默认要求人工确认后再进入执行队列。
- 支持模型列表、激活、回滚、批量推理和发布阶段管理。
- 支持将标注结果转化为训练输入，形成从反馈到训练的闭环。

### 3.5 主动学习与标注闭环

- 系统应能识别高不确定性的帧并进入标注队列。
- 标注队列需要支持查看、标注、跳过、统计和图片访问。
- 已标注样本需要能够触发增量训练任务创建。
- 标注和训练的联动当前以“后端能力 + 模型页标签入口”为主，而不是独立前端主页面。

### 3.6 Dashboard 交互范围

当前前端主视图范围如下：

- 总览页：展示系统状态、摄像头状态、告警摘要和待办信息。
- 摄像头页：展示摄像头列表、控制和详情入口。
- 摄像头详情页：展示单路摄像头画面、状态和诊断信息。
- 告警页：展示告警列表、筛选、工作流操作和回放详情。
- 模型页：通过标签页聚合基线、训练与评估、模型与发布、A/B 对比、标注队列和阈值预览。
- 系统页：通过标签页聚合系统概览、配置管理、数据备份、审计日志、降级历史、音频告警、用户管理和存储清理。

### 3.7 运维与管理能力

- 支持认证、会话、限流和安全响应头。
- 支持配置重载、数据库备份和旧数据清理。
- 支持推理记录、审计日志、任务状态和运行指标查看。
- 支持全局降级事件展示与历史查询。
- 支持用户管理和角色区分。

## 4. 默认运行路径与非默认能力

### 4.1 默认运行路径

以下能力属于当前默认部署中的主路径：

- go2rtc 集成。
- Simplex 安全通道。
- 健康检查与状态汇总。
- 漂移检测。
- 环形缓冲与录像回放链路。
- 人工确认后的训练作业执行模型。

### 4.2 非默认但已存在的能力

以下能力已经在代码或配置中出现，但默认不作为主路径：

- 开放词汇分类器。
- 分割器。
- 跨摄像头关联。
- 自动重训练。

文档、演示和对外介绍应明确把这些能力标注为“可选”“扩展”或“实验性接入”，避免误导为默认启用。

## 5. 非功能要求

### 5.1 稳定性

- 单路摄像头失败不应导致全局停止。
- 视频流路径不应阻塞普通 API 请求。
- 中间件实现必须兼容长连接与流式响应。
- 默认运行路径中的关键组件需要具备故障隔离和降级能力。

### 5.2 可追溯性

- 告警、任务、训练作业、用户行为和关键状态切换需要可追溯。
- 关键运行信息需要通过 API、日志或前端页面可见。
- 回放、推理记录和审计日志应支持问题复盘。

### 5.3 可运维性

- 系统需要支持配置重载、备份、清理和运行状态诊断。
- 目录结构应允许长期保存基线、模型、录像、推理记录和备份。
- 当前工作区应能以默认配置直接启动本地样例环境。

### 5.4 可配置性

- 主要运行参数通过 YAML 配置。
- 新增配置项时必须同时更新 schema 与默认值。
- 对默认关闭的模块，应能通过配置明确开启，而不是通过隐式代码路径生效。

### 5.5 前后端边界清晰

- 后端提供 API、状态模型和后台执行能力。
- 前端只对已存在的 API 和 WebSocket 主题做交互封装。
- 不应把“有后端 API”直接写成“已有独立前端页面”。

## 6. 当前明确不包含的表达方式

以下内容不应继续作为当前需求文档的写法：

- 未经验证的精确模块数量、测试数量、页面数量。
- 将关闭状态的能力描述为默认主流程。
- 将后端 API 能力直接等价为前端独立页面。
- 将历史愿景清单直接当作当前交付范围。
- 将尚未接入启动链或默认流程的能力描述为“当前已生效”。

## 7. 后续维护原则

- 需求文档描述“当前交付范围”和“明确的默认/非默认能力”。
- 架构文档描述“当前实现方式”和运行时结构。
- README 描述“如何理解、启动和浏览仓库”。
