# Argus 需求文档

## 1. 项目概述

### 1.1 背景
核电站关键区域（反应堆厂房、安全壳内、管廊、乏燃料水池周边等）实行严格的异物防控（Foreign Object Exclusion, FOE）。任何非预期物体（墙皮脱落、保温层掉落、遗留工具、杂物等）都可能对设备安全构成威胁。

### 1.2 目标
通过固定摄像头 + 深度学习异常检测，实现：
- 实时监控关键区域，自动发现异常物体
- 分级报警，推送到控制室
- 最小化误报率（核电站环境下报警必须有意义）
- 7×24 小时不间断运行

### 1.3 系统名称
Argus（阿耳戈斯，希腊神话中的百眼巨人）

## 2. 功能需求

### 2.1 视觉检测管线

#### FR-001 摄像头采集
- 支持 RTSP、USB、视频文件三种输入源
- 支持配置目标帧率（帧率抽取，默认 5FPS）
- 支持配置分辨率
- 断线自动重连，指数退避，可配置最大重试次数

#### FR-002 预筛阶段（MOG2 背景减除）
- 使用 MOG2 算法建模背景，检测前景变化
- 无变化的帧直接跳过，节约后续算力
- 阴影检测开关（避免阴影触发假阳性）
- 变化比例阈值可配置
- **辐射环境适配**：前置中值滤波消除 CMOS 辐射椒盐噪声

#### FR-003 人员过滤（YOLO）
- 使用 YOLO 检测画面中的人员
- 两种模式：遮罩（将人员区域涂黑）或跳帧（整帧跳过）
- YOLO 不可用时优雅降级（跳过人员过滤，不崩溃）

#### FR-004 异常检测（Anomalib）
- 使用 PatchCore 或 EfficientAD 算法
- 只需要"正常"图像训练，不需要异常样本标注
- 输出：异常分数（0-1）+ 逐像素异常热力图
- 冷启动回退：无模型时使用 SSIM 占位
- **模型热更新**：运行中原子替换模型，不中断推理

#### FR-005 防背景吸收
- **心跳全帧检测**：每 N 帧（默认 150 帧 ≈ 30 秒）强制跳过 MOG2，直接送 Anomalib 检测
- **异常状态锁定**：高置信度异常（>0.85）触发锁定，后续帧持续绕过 MOG2 直到异常消失或操作员清除

### 2.2 区域管理

#### FR-006 检测区域（Include Zone）
- 多边形定义，限定检测范围
- 多个检测区域可共存，各自独立评估和报警
- 每个区域可设置独立的异常阈值和优先级

#### FR-007 排除区域（Exclude Zone）
- 多边形定义，该区域内的任何变化都不检测
- 典型用途：旋转风扇、闪烁指示灯、常开门等
- 排除区域优先级高于检测区域（重叠时排除生效）

#### FR-008 区域热更新
- 区域增删不需要重启系统
- 通过 Dashboard 或 API 实时修改

### 2.3 报警系统

#### FR-009 四级报警分级
- INFO（0.50-0.70）：仅记录
- LOW（0.70-0.85）：Dashboard 通知
- MEDIUM（0.85-0.95）：推送控制室
- HIGH（>0.95）：全通道报警

#### FR-010 时间窗滤波
- 异常必须在连续 N 帧（默认 3 帧）中持续出现才触发报警
- 两次检测间隔超过 M 秒（默认 10 秒）则重置计数

#### FR-011 重复抑制
- 同一区域在 T 秒内（默认 300 秒）不重复报警

#### FR-012 报警分发
- 数据库持久化（所有报警）
- Webhook HTTP POST（可配置 URL，用于对接电厂系统）
- 快照和异常热力图自动保存到磁盘

#### FR-013 报警反馈
- 操作员可确认报警
- 操作员可标记误报（用于后续模型改进）

### 2.4 多摄像头支持

#### FR-014 线程池管理
- 每个摄像头运行在独立线程中
- 单个摄像头故障不影响其他摄像头
- 支持运行时独立启停单个摄像头

### 2.5 Dashboard 仪表盘

#### FR-015 系统概览
- 系统状态指标卡片
- 摄像头网格（在线状态、帧数、延迟）
- 最近报警列表

#### FR-016 摄像头页面
- 每个摄像头的详细统计
- 跳过率、分析帧数、报警数

#### FR-017 报警管理
- 分页报警列表
- 按严重级别/摄像头筛选
- 一键确认/标记误报

#### FR-018 区域编辑器
- 摄像头截图上 Canvas 画多边形
- 绿色 = 检测区域，红色 = 排除区域
- 保存后即时生效（不重启）

#### FR-019 配置控制台
- 阈值调整（实时生效）
- 摄像头重启
- 清除异常锁定
- 模型热更新触发

#### FR-020 JSON API
- `/api/system/health` — 系统健康
- `/api/alerts/json` — 报警数据
- `/api/cameras/json` — 摄像头状态

### 2.6 基线与训练

#### FR-021 基线管理
- 版本化存储（v001, v002...）
- 采集脚本（指定源、数量、间隔）
- 旧版本自动清理

#### FR-022 模型训练
- Anomalib PatchCore / EfficientAD 训练脚本
- 导出到 OpenVINO / ONNX 格式

#### FR-023 误报反馈闭环
- 误报快照自动导出到基线目录
- 下次重训练时纳入，使模型学会忽略类似模式

### 2.7 部署

#### FR-024 容器化
- Docker 多阶段构建
- docker-compose 一键部署
- 数据卷持久化
- 资源限制（CPU/内存）

## 3. 非功能需求

### NFR-001 延迟
- 从帧采集到报警触发 < 500ms

### NFR-002 可用性
- 7×24 小时不间断运行
- 单摄像头故障不影响全局
- 自动重连、自动恢复

### NFR-003 误报率
- 时间窗滤波 + 区域掩码 + 辐射去噪，目标 < 1 次/天/摄像头

### NFR-004 安全
- Docker 非 root 用户运行
- 健康检查端点

### NFR-005 可维护性
- Pydantic 类型安全配置
- 结构化日志（structlog）
- 83+ 单元测试

## 4. 约束条件

- 边缘部署：Intel NUC 或 NVIDIA Jetson Orin NX
- 每节点 2-4 路摄像头
- Python 3.11+
- 核电站网络环境（可能无外网访问）
